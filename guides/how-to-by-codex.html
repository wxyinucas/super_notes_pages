静态本地网站使用指南 ​

这份指南说明如何维护 journal-web/ 这个本地 VitePress 网站。它的目标是：内容可以拆成小 Markdown 文件，网站可以聚合展示，并保留公式、目录、旁注、分栏和中文排版能力。

先记住三件事 ​

  1. 根目录 super_notes/ 不直接做网站。它保留给协作契约、项目说明、agent 配置和其他非网页文件。
  2. 网站只放在 journal-web/ 里。运行、构建、主题、导航都从这里管理。
  3. 内容采用“一个站点，多套内容目录”。也就是说，不为每个项目开一个新网站，而是在同一个 VitePress 站点下拆出 entries/、projects/、readings/、guides/ 等目录。

快速运行 ​

进入网站目录：

bash
cd journal-web

安装依赖：

bash
npm install

启动本地预览：

bash
npm run dev

默认地址是：

text
http://127.0.0.1:5173/

构建静态网站：

bash
npm run build

预览构建结果：

bash
npm run preview

各部分的职责：

  • journal-web/index.md：网站首页。
  • journal-web/entries/：日志、阶段性思考、正式笔记。
  • journal-web/readings/：读书笔记、摘录、批注。
  • journal-web/projects/：长期项目记录、设计备忘、阶段复盘。
  • journal-web/snippets/：组件示例、写作片段、短说明。
  • journal-web/guides/：网站使用指南、写作约定、维护说明。
  • journal-web/public/：公开静态资源，例如图片、字体、图标。
  • journal-web/.vitepress/config.mjs：站点标题、顶部导航、侧边栏、右侧目录、Markdown 插件。
  • journal-web/.vitepress/theme/style.css：字体、颜色、正文宽度、旁注布局。
  • 各内容域的 index.md：入口和聚合页。
  • 各内容域的 template.md：写作骨架和可复制模板。
  • journal-web/.vitepress/theme/components/：全局 Vue 组件，例如 <MarginNote>、<QuoteBlock>、<SplitNote>、<StaticOutline> 和 <Columns>。

内容组织原则 ​

建议按“内容域”拆一级目录。内容域不是单独的网站，而是同一个网站里的不同资料区。

推荐结构：

text
journal-web/
├── entries/          # 日志、日常笔记、阶段性思考
├── readings/         # 读书笔记、摘录、批注
├── projects/         # 具体项目的长期记录
├── math/             # 数学、公式推导、技术札记
├── snippets/         # 写作组件和短片段
└── guides/           # 本网站自己的使用说明

每个内容域最好都有一个 index.md 和一个 template.md：

  • index.md：作为该区域的引导页和目录页。
  • template.md：放该区域的写作骨架和可复制模板。

例如：

text
projects/
├── index.md
├── template.md
├── journal-system.md
└── ai-agent-contract.md

对应访问路径：

  • projects/index.md -> /projects/
  • projects/template.md -> /projects/template
  • projects/journal-system.md -> /projects/journal-system
  • readings/book-a.md -> /readings/book-a

保持一个简单规则：目录页负责聚合，模板页负责写法，普通文章负责写内容，导航只放高频入口。

嵌套内容结构 ​

文件夹可以继续向下嵌套。Web 上的呈现规则很直接：目录里的 index.md 会成为这个目录的入口页，普通 .md 文件会成为独立页面。

以“读书 -> 一本书 -> 章节笔记”为例：

text
readings/
├── index.md
├── template.md
└── books/
    └── book-a/
        ├── index.md
        ├── template.md
        ├── chapter-01.md
        ├── chapter-02.md
        └── notes/
            └── concept-x.md

对应访问路径：

  • readings/index.md -> /readings/
  • readings/books/book-a/index.md -> /readings/books/book-a/
  • readings/books/book-a/chapter-01.md -> /readings/books/book-a/chapter-01
  • readings/books/book-a/notes/concept-x.md -> /readings/books/book-a/notes/concept-x

维护建议：

  • 每个有稳定含义的目录都放一个 index.md，让它成为本层的引导页。
  • 每个需要长期复用写法的内容层级都可以放一个 template.md，例如某个内容域、某本书、某个项目。
  • 顶部导航只放一级内容域，例如“日志”“读书”“项目”“指南”。
  • 侧边栏只放稳定且高频的入口，可以嵌套一两层，例如“读书 -> 某本书”。
  • 更深的内容不要全塞进导航栏，交给每层 index.md 做局部目录。

一个常见写法是让某本书的 index.md 承担“书内首页”：

md
---
title: 某本书
---

# 某本书

## 章节笔记

- [第一章](/readings/books/book-a/chapter-01)
- [第二章](/readings/books/book-a/chapter-02)

## 概念索引

- [概念 X](/readings/books/book-a/notes/concept-x)

这样文件系统可以很深，但读者在 Web 上看到的是一层一层的入口页，而不是一条过长的侧边栏。

目录页驱动页脚导航 ​

本项目会自动读取深层目录里的 index.md 链接顺序，为这些链接指向的页面生成底部页脚导航。页脚包含三栏：上一篇、目录页、下一篇。这意味着书籍、项目或月份页面不必把所有子页面塞进侧边栏，只要在上级目录页里维护一个有序链接列表即可。

例如某本书的 index.md：

md
# 某本书

- [第一章](./chapter-01)
- [第二章](./chapter-02)
- [第三章](./chapter-03)

访问 /readings/books/book-a/chapter-02 时，底部会自动出现：

  • 上一篇：第一章
  • 目录页：某本书
  • 下一篇：第三章

首篇和末篇会回到该目录的 index.md。如果某一页需要特殊跳转，可以在该页 frontmatter 中手写 prev 或 next 覆盖自动结果：

md
---
next:
  text: "附录"
  link: "/readings/books/book-a/appendix"
---

拼接页面 ​

如果想把多个 Markdown 文件在 Web 上拼成一个页面，可以使用 VitePress 的 Markdown include 语法。它是构建期拼接：开发和构建时把被引用文件插入当前页面，最终仍生成一个静态 HTML 页面。

推荐把只用于拼接的片段放进局部目录：

text
readings/books/book-a/
├── index.md
├── combined.md
└── parts/
    ├── A.md
    └── B.md

combined.md 可以这样写：

md
---
title: 合并阅读页
---

# 合并阅读页

<!-- @include: ./parts/A.md -->

<!-- @include: ./parts/B.md -->

这样访问 /readings/books/book-a/combined 时，会看到 A.md 和 B.md 前后拼成的页面。

也可以引用整个 journal-web/ 下的绝对路径：

md
<!-- @include: @/readings/books/book-a/parts/A.md -->

更细的拼接方式：

md
<!-- @include: ./parts/A.md#region-name -->

<!-- @include: ./parts/A.md{10,30} -->

注意事项：

  • 被整体 include 的 Markdown 文件如果有 frontmatter，VitePress 会在拼接时去掉这段 frontmatter。
  • 如果父页面已经有 # 一级标题，被拼接文件里最好从 ## 开始，避免一个页面出现多个竞争性的一级标题。
  • 如果 A.md 和 B.md 只是片段，不要把它们加入侧边栏或聚合页。
  • 如果 A.md 和 B.md 同时也是独立页面，也可以被 include；只是 Web 上会同时存在独立页面和合并页面，需要注意重复内容和标题层级。

新建一篇文章 ​

以新建一篇 2026 年日志为例。

第一步，创建文件：

text
journal-web/entries/2026/my-new-note.md

第二步，写 frontmatter 和正文：

md
---
title: 我的新日志
---

# 我的新日志

正文从这里开始。

第三步，把它加入聚合页。编辑 journal-web/entries/index.md 或 journal-web/entries/2026/index.md：

md
- [我的新日志](/entries/2026/my-new-note)

第四步，判断是否需要加入侧边栏。高频入口可以加入 .vitepress/config.mjs 的 themeConfig.sidebar；临时文章不一定要加，放在聚合页里即可。

新建一个月度日志页 ​

月度日志适合用一个目录承接，例如：

text
journal-web/entries/2026/06_jun/index.md

月度首页可以在标题后放一个只显示本月的月历。MonthCalendar 使用 year 和 month 作为输入，周从周日开始：

md
---
title: 2026_June
---

# 六月: 考试前最后的学期

<MonthCalendar year="2026" month="6" />

日记条目可以继续放在这个月目录下，再从月度首页手动整理入口。这样 Web 上的呈现单位是“月”，但具体写作仍可以保持每天或每篇一份 Markdown。

新建一个内容域 ​

以新增 readings/ 读书笔记区为例。

第一步，新建目录和首页：

text
journal-web/readings/index.md

index.md 可以这样写：

md
---
title: 读书笔记
---

# 读书笔记

## 书目

- [第一本书](/readings/book-a)

第二步，新建具体文章：

text
journal-web/readings/book-a.md

第三步，在顶部导航里加入口。编辑 .vitepress/config.mjs：

js
nav: [
  { text: '日志', link: '/entries/' },
  { text: '读书', link: '/readings/' },
  { text: '片段', link: '/snippets/' },
  { text: '指南', link: '/guides/' }
]

第四步，在侧边栏里加分组：

js
sidebar: [
  {
    text: '读书',
    items: [
      { text: '读书总览', link: '/readings/' },
      { text: '第一本书', link: '/readings/book-a' }
    ]
  }
]

顶部导航适合放一级内容域，数量控制在 4 到 6 个以内。侧边栏适合放更细的层级，例如年份、主题、项目名或书名。

链接和路径规则 ​

当前开启了 cleanUrls: true，链接不用写 .html。

常见映射：

text
journal-web/index.md                         -> /
journal-web/entries/index.md                 -> /entries/
journal-web/entries/2026/first-note.md       -> /entries/2026/first-note
journal-web/guides/how-to-by-codex.md        -> /guides/how-to-by-codex

写站内链接时推荐用绝对路径：

md
[日志总览](/entries/)
[静态本地网站使用指南](/guides/how-to-by-codex)

这样移动当前文件时，链接不容易失效。

顶部导航和侧边栏 ​

顶部导航和侧边栏都在 .vitepress/config.mjs 的 themeConfig 里。

顶部导航：

js
nav: [
  { text: '日志', link: '/entries/' },
  { text: '片段', link: '/snippets/' },
  { text: '指南', link: '/guides/' }
]

侧边栏：

js
sidebar: [
  {
    text: '日志',
    items: [
      { text: '总览', link: '/entries/' },
      { text: '第一篇示例', link: '/entries/2026/first-note' }
    ]
  },
  {
    text: '指南',
    items: [
      { text: '指南总览', link: '/guides/' },
      { text: '静态本地网站使用指南', link: '/guides/how-to-by-codex' }
    ]
  }
]

侧边栏可以嵌套：

js
{
  text: '日志',
  items: [
    { text: '总览', link: '/entries/' },
    {
      text: '2026',
      items: [
        { text: '第一篇示例', link: '/entries/2026/first-note' }
      ]
    }
  ]
}

维护建议：

  • nav 放大类入口，例如“日志”“读书”“项目”“指南”。
  • sidebar 放常用文章和稳定目录。
  • index.md 放完整索引。内容多起来以后，不要把每篇文章都塞进 sidebar。

页面内目录和跳转 ​

右侧浮动目录由自定义组件 <StaticOutline> 显示，并保持 sticky 浮动。标题层级仍参考 .vitepress/config.mjs 中的 VitePress outline 配置：

js
outline: {
  level: [2, 4],
  label: '目录'
}

常见改法：

  • 只显示二级标题：level: [2, 2]。
  • 当前显示二到四级标题：level: [2, 4]。
  • 显示二到六级标题：level: 'deep'。
  • 改目录标题：label: '本文目录'。

目录位置、sticky 偏移和最大高度优先在 theme.config.toml 的 [layout] 中调整。

页面内跳转直接使用标题锚点：

md
## 一个公式段落

[跳到公式段落](#一个公式段落)

标题较长时，建议手动指定短锚点：

md
## 一个很长的标题 {#short-anchor}

[跳到这里](#short-anchor)

也可以在文章开头写一个局部目录：

md
## 快速跳转

- [背景](#background)
- [推导](#derivation)
- [结论](#result)

## 背景 {#background}

## 推导 {#derivation}

## 结论 {#result}

具体内容格式的调整 ​

公式 ​

全站启用了 KaTeX，可以直接在 Markdown 中写公式。

行内公式：

md
这是行内公式：$E = mc^2$。

块级公式：

md
$$
\int_a^b f(x)\,dx = F(b) - F(a)
$$

也支持 \(...\) 和 \[...\] 形式。

旁注 ​

轻量旁注使用 <MarginNote>。

单段旁注：

md
<MarginNote markdown="这里支持 **Markdown** 和 $a^2+b^2=c^2$。" />

多段旁注：

md
<MarginNote label="旁注">

这里可以写多段 Markdown。

</MarginNote>

当前布局策略：

  • 小屏和普通宽度下，旁注进入正文流，避免和右侧目录重叠。
  • 超宽屏下，旁注浮到正文右侧、目录左边。

对应样式在 .vitepress/theme/style.css：

css
@media (min-width: 1536px) {
  .VPDoc:has(.margin-note) .content-container {
    max-width: 688px !important;
  }

  .vp-doc .margin-note {
    float: right;
    clear: right;
    width: 12.5rem;
    margin-right: -13.5rem;
    margin-left: 1rem;
  }

  .VPDoc:has(.margin-note) .aside {
    transform: translateX(5rem);
  }
}

调节方向：

  • 想让旁注更宽：增大 .margin-note 的 width。
  • 想让旁注离正文更远：增大 .margin-note 的 margin-left。
  • 想让正文更宽：增大 .content-container 的 max-width。
  • 想让旁注更早浮动：降低媒体查询里的 1536px，但要注意和右侧目录重叠。

分栏 ​

普通分栏使用 <Columns>：

md
<Columns :count="2">

左栏内容。

右栏内容。

</Columns>

这个组件适合短片段、对照说明、摘要和 TODO。移动端会自动回到单栏。

读书笔记中的“正文 0.7 + 批注 0.3”不建议用 <Columns> 硬做。它更像一种和正文同等重要的双栏阅读模式，应使用 <SplitNote>：

md
<SplitNote>
  <template #main>

这里是正文，占 70%。

  </template>

  <template #aside>

这里是同等重要的读后感、批注、摘录评论，占 30%。

  </template>
</SplitNote>

建议语义区分：

  • <MarginNote>：轻量旁注，服务于正文。
  • <QuoteBlock>：摘录原文、论文段落或材料片段。
  • <Columns>：普通视觉分栏，不区分主次。
  • <SplitNote>：正文和批注并列，二者都是核心阅读对象。
  • <SplitNoteInsert>：把批注轻量绑定到标题锚点，并自动把目标标题之后的一段正文重排为 SplitNote。
  • <MonthCalendar>：月度日志首页的月历，只显示指定年月。

标题锚定式插入：

md
## 理解焦虑 {#anxiety}

这里是正文。

<SplitNoteInsert target="anxiety">

这里是对应本节的批注。

</SplitNoteInsert>

target 推荐绑定手写锚点，例如 {#anxiety}。如果没有写 target，组件会尝试绑定到它前面最近的带 id 标题；也可以用 level="h2" 限定要寻找的标题层级。默认情况下，插入块本身就是截断点：目标标题之后、插入块之前的正文进入主栏，它里面的内容进入侧栏。

如果想把所有笔记集中放在文末，使用 scope="section"：

md
## 理解焦虑 {#anxiety}

这里是正文。

## 焦虑和目标 {#goals}

这里是下一节正文。

<SplitNoteInsert target="anxiety" scope="section">

这里是“理解焦虑”这一节的批注。

</SplitNoteInsert>

scope="section" 会保留目标标题在原始位置，并从目标标题后开始，自动截到下一个同级或更高级标题之前。也可以用 until 手动指定结束锚点，适合章节开头、跨小节或标题层级不规整的情况：

md
<SplitNoteInsert target="chapter-intro" until="first-section">

这里是章前导语的批注。

</SplitNoteInsert>

优先级是：until 优先，其次 scope="section"，最后回退到默认插入点截断。

摘录块 ​

摘录使用 <QuoteBlock>。它只负责呈现原始材料，不承担批注职责。

md
<QuoteBlock source="作者《书名》，章节或页码">

这里放原文摘录。可以包含 **Markdown** 和公式 $a^2+b^2=c^2$。

</QuoteBlock>

读书笔记优先组合 <QuoteBlock>、<SplitNote> 和 <MarginNote>。暂时不新增 <ReadingNote> 组件；ReadingNote 先作为一种写作范式维护。

页面宽度和留白 ​

页面布局优先在 theme.config.toml 调整。npm run build、npm run dev 和 npm run preview 会先运行生成脚本，把配置写入 .vitepress/theme/generated-vars.css。

正文基础样式：

toml
[layout]
docFontSize = "17px"
docLineHeight = "1.75"
headingPrefixWidth = "0.22rem"
headingPrefixTop = "0.35em"
headingPrefixBottom = "0.2em"
headingPaddingLeft = "1rem"
outlineSticky = "true"
outlineTop = "calc(var(--vp-nav-height) + 2rem)"
outlineMaxHeight = "calc(100vh - var(--vp-nav-height) - 4rem)"

旁注页面的正文宽度：

toml
[layout]
marginNoteBreakpoint = "1536px"

组件间距和摘录字号：

修改前先判断你想解决哪类问题：

  • 内容太窄：增大 .content-container 的 max-width。
  • 页面太挤：增大正文、旁注、目录之间的间距。
  • 目录离正文太近：调整 VitePress 的 .aside、.VPDocAside 或 .VPDoc 相关样式。
  • 旁注和目录重叠：增大断点，缩小旁注，或让旁注回到正文流。

建议保留三种屏幕状态：

  • 小屏：单栏阅读。
  • 中屏：正文 + 右侧目录。
  • 大屏：正文 + 旁注 + 右侧目录。

字体和颜色 ​

字体和颜色优先改 theme.config.toml，不要手改 .vitepress/theme/generated-vars.css。生成文件是构建脚本的产物，下次运行命令会被覆盖。

当前字体设置：

toml
[fonts]
body = 'ui-serif, Georgia, Cambria, "Times New Roman", "KaiTi", "STKaiti", "Kaiti SC", serif'
heading = 'ui-serif, Georgia, Cambria, "Times New Roman", "STXingkai", "Xingkai SC", "华文行楷", serif'
strong = '"Kaiti SC", "STKaiti", "KaiTi", ui-serif, Georgia, Cambria, "Times New Roman", serif'
emphasis = 'ui-serif, Georgia, Cambria, "Times New Roman", "KaiTi", "STKaiti", "Kaiti SC", serif'
mono = '"SFMono-Regular", Consolas, "Liberation Mono", Menlo, monospace'

这里使用“英文字体在前，中文字体在后”的顺序。浏览器会先用前面的字体显示英文和数字；这些字体没有中文字形时，中文才会落到后面的中文字体。

效果：

  • 正文中文优先使用楷体。
  • h1 到 h5 的中文标题优先使用行楷。
  • **加粗** 优先使用更重的楷体，并在深色模式下更亮。
  • *斜体* 使用轻微斜体、柔和下划线和底色，适合作为读书标记式强调。
  • 英文和数字仍保留西文字体。

标题颜色可以分别配置：

toml
[colors.light]
h1 = "#173b31"
h2 = "#205646"
h3 = "#2d6354"
h4 = "#365f55"
h5 = "#4a685f"
strong = "#16231e"
italic = "#2a6555"
italicUnderline = "#8bbdac"
italicBackground = "#edf5f1"

正文颜色、品牌色和旁注颜色也在同一节：

toml
[colors.light]
brand1 = "#2f6f5e"
brand2 = "#3b826f"
brand3 = "#6aa58f"
body = "#27332f"
marginNoteText = "#46564f"

深色模式对应配置在 [colors.dark]。如果只改浅色而忘了深色，切换主题时会出现色彩不一致。

如果本机没有 STXingkai、Xingkai SC、华文行楷、KaiTi 或 STKaiti，浏览器会继续 fallback。若希望不同机器显示完全一致，需要把字体文件放进 public/fonts/，再用 @font-face 注册。

手动刷新主题变量：

bash
npm run theme:generate

发布到 GitHub Pages ​

GitHub Free 计划下，private repository 不能直接启用 GitHub Pages；页面会提示需要升级或把仓库改成 public。因此当前采用“双仓库”发布：

  • 本仓库保持 private，用来写作、预览和构建。
  • 另建一个 public 仓库，只保存静态构建产物，并从那里启用 GitHub Pages。

发布构建不做内容过滤：journal-web/ 中会被 VitePress 构建出来的页面，都可能进入 public 发布仓库和公开 Pages。

本地和远程的分工 ​

本地写作仍然使用原来的命令：

bash
cd journal-web
npm run dev
npm run build

推送到 GitHub 后，.github/workflows/deploy-pages.yml 会自动执行：

  1. 在 journal-web/ 下运行 npm ci。
  2. 运行 npm run build。
  3. 把 journal-web/.vitepress/dist 复制到 public 发布仓库。
  4. 提交并推送 public 发布仓库的发布分支。

public 发布仓库再通过自己的 Pages 设置对外提供静态网站。

GitHub 端设置 ​

第一次启用时，需要准备一个 public 发布仓库。推荐使用专用空仓库，例如：

text
wxyinucas/super_notes-pages

如果使用普通项目 Pages，最终地址通常是：

text
https://用户名.github.io/super_notes-pages/

如果使用用户站点仓库，仓库名必须是：

text
用户名.github.io

最终地址是：

text
https://用户名.github.io/

在 public 发布仓库中设置 Pages：

  1. 仓库必须是 public，除非 GitHub 账号有支持 private Pages 的付费计划。
  2. 进入 Settings -> Pages。
  3. 在 Build and deployment 里选择 Source: Deploy from a branch。
  4. 分支选择发布分支，默认是 main。
  5. 目录选择 / (root)。

在 private 写作仓库中设置 Actions secret 和 variables：

text
PAGES_DEPLOY_TOKEN=能写 public 发布仓库的 token
PAGES_REPO=wxyinucas/super_notes-pages
PAGES_BRANCH=main

PAGES_DEPLOY_TOKEN 建议使用 fine-grained personal access token，只授予 public 发布仓库的 Contents: Read and write 权限。

PAGES_REPO 和 PAGES_BRANCH 放在 Settings -> Secrets and variables -> Actions -> Variables。PAGES_DEPLOY_TOKEN 放在同一页的 Secrets。

然后 push 到 private 写作仓库的 main，或在 Actions 页面手动运行：

text
Publish Journal Web to Public Pages Repo

VitePress base ​

workflow 会根据 public 发布仓库名自动设置 VitePress 的 base：

  • 发布到 wxyinucas/super_notes-pages 时，默认 base 是 /super_notes-pages/。
  • 发布到 wxyinucas.github.io 时，默认 base 是 /。

如果使用自定义域名，或希望自己指定路径，可以在 private 写作仓库的 Actions Variables 中添加：

text
VITEPRESS_BASE=/

也可以设为其他路径，例如：

text
VITEPRESS_BASE=/notes/

本地没有设置 VITEPRESS_BASE 时，VitePress 仍使用 /，不会影响 npm run dev 和本地预览。

本地 / 远端发布开关 ​

本地写作和预览默认显示全部内容：

bash
npm run dev
npm run build

公开发布时，workflow 会设置：

text
JOURNAL_BUILD_TARGET=public

此时 VitePress 会排除标记为本地-only 的 Markdown 页面，并从顶部导航和侧边栏里移除指向这些页面的入口。

推荐在“概念上较大”的目录入口页上打标，例如一本书、一年日记或一个长期项目的 index.md：

md
---
title: 某本书
publish: local
---

# 某本书

这个标记会被该目录下所有页面继承。例如：

text
readings/books/book-a/index.md       # publish: local
readings/books/book-a/chapter-01.md  # 公开构建时自动排除
readings/books/book-a/notes/a.md     # 公开构建时自动排除

如果只想隐藏单篇文章，也可以在该文章自己的 frontmatter 中写：

md
---
title: 某篇日志
public: false
---

当前支持的写法：

  • publish: local
  • publish: private
  • public: false
  • visibility: local
  • label: local-only
  • labels: [local-only]
  • tag: local-only
  • tags: [local-only]

发布状态更推荐用 publish 字段；tag 和 label 适合临时兼容，但容易和内容分类混在一起。

如果某个子页面需要覆盖父级目录的本地-only 标记，可以在子页面写：

md
---
title: 可公开的摘要
publish: public
---

不写发布标记时，默认视为可公开。这个默认值适合当前“需要公开时能公开”的阶段；如果以后要改成默认不公开，再把策略切换为只有 publish: public 才进入远端构建。

本地模拟公开构建：

bash
npm run build:public

注意：顶部导航和侧边栏会自动过滤，但正文里的手写链接不会自动删除。如果公开页面继续链接到本地-only 页面，远端会出现无效链接，也可能泄露标题线索。稳妥做法是：把整个上层目录标记为 publish: local，或在公开索引里不要链接本地-only 内容。

公开边界 ​

当前版本只做基础 frontmatter 过滤，不做复杂内容改写、自动索引重算或摘录脱敏。需要临时分享时，可以打开 Pages；暂时不想公开时，可以在 public 发布仓库的 Settings -> Pages 中关闭 Pages，或停用 private 写作仓库里的 workflow。

已经发布过的内容不应视为可完全收回，浏览器缓存、搜索引擎缓存和他人保存不受本站控制。后续如果要做选择性公开，再引入 public: false 或 public: true 的发布构建脚本。

常用维护清单 ​

新增一篇普通文章： ​

  1. 创建 Markdown 文件。
  2. 写 frontmatter 和一级标题。
  3. 把链接加入对应内容域的 index.md。
  4. 高频文章再加入 sidebar。
  5. 运行 npm run build 检查。

新增一个内容域： ​

  1. 创建一级目录。
  2. 创建该目录的 index.md。
  3. 创建该目录的 template.md。
  4. 在 nav 增加入口。
  5. 在 sidebar 增加分组。
  6. 在首页或相关聚合页增加链接。
  7. 运行 npm run build 检查。

调整布局： ​

  1. 先确定目标：正文宽度、旁注宽度、目录位置，还是移动端表现。
  2. 优先修改 theme.config.toml。
  3. 分别检查小屏、中屏、大屏。
  4. 运行 npm run build。

调整导航： ​

  1. 修改 .vitepress/config.mjs。
  2. 确认链接路径不带 .html。
  3. 打开页面检查顶部导航、侧边栏和右侧目录。
  4. 运行 npm run build。
